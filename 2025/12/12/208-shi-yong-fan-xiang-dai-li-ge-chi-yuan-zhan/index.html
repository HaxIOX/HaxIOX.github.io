使用反向代理隔离源站
2025-12-28 21:51:43 #反向代理

背景

想搞一下站点，赚点小钱钱，又怕有大佬攻城，想藏一下源ip，但是使用cloudflare代理的效果不太好，于是去求助伟大的gpt，向G老师学习

“用入口反向代理做了源站 IP 隔离，源站只对白名单入口开放，不直接暴露公网。”

很好，反代机被判定为滥用钓鱼给暂停了，简单的原因分析以及解决方案给我放出来了

您好，
可能发生的过程：服务器遭到自动化扫描/攻击，并被第三方滥用，触发了钓鱼相关风控。

我应对的方案：
1）反代配置固定上游，未知 Host 直接丢弃，避免被当作开放代理；
2）防火墙仅开放 80/443，SSH 仅允许我的管理 IP，并更换所有密码/密钥；
3）我同意你们为该服务进行系统重装以解除暂停。

请协助在重装完成后解除暂停，谢谢。

环境

  • 一台源站附近的代理服务器(目前用低配置1c1g的美国优化机测试)

    代理服务器放在「离源站最近」的地区，而不是离用户最近。

  • 一台源服务器

前置知识

iptables

“iptables 是 Linux 内核里的“网络安检规则表”

所有进出服务器的数据包都经过iptables的检查

iptables 在系统里的真实位置

数据流不是直接到 Nginx / Docker 的，而是：

网卡
 ↓
Linux 内核网络栈
 ↓
iptables
 ↓
Nginx / SSH / Docker

核心概念

  • 表-filter表-放行/拒绝/丢弃

  • 链

    INPUT ——「进本机」

    OUTPUT ——「本机发出去」

    FORWARD ——「转发」

    典型场景：

    • Docker 容器
    • NAT 网关
    • 虚拟机桥接

  • 规则

    ptables 是 从上往下 一条一条看：

    第一条不匹配 → 看第二条
匹配了 → 立刻执行动作 → 停止

    动作包括：

    • ACCEPT（放行）
    • DROP（直接丢）
    • REJECT（丢并告诉对方）
    • RETURN（回到上一层链）

思路

  • 封锁源站防火墙，只对代理ip开放
  • 代理服务器将

配置

源站

源站仅接受代理机的访问

在源站上（别把自己 SSH 锁死了，先放行 22 的你的管理 IP）：

# 先允许 SSH（把 YOUR_IP 换成你自己的公网IP；没有固定IP就先临时放开22）
iptables -A INPUT -p tcp --dport 22 -j ACCEPT

# 允许已建立连接
iptables -A INPUT -m conntrack --ctstate ESTABLISHED,RELATED -j ACCEPT

# 只允许代理访问静态站 80
iptables -A INPUT -s PROXY_IP -p tcp --dport 80 -j ACCEPT

# 只允许代理访问 docker 3000
iptables -A INPUT -s PROXY_IP -p tcp --dport 3000 -j ACCEPT

# 回环
iptables -A INPUT -i lo -j ACCEPT

# 默认拒绝（放最后执行，避免断连）
iptables -P INPUT DROP

源站 443 不需要开放。

确认源站服务监听没问题

源站上跑：

ss -lntp | egrep ':80|:3000'

你应该看到：

  • :80 在监听（静态站）
  • :3000 在监听（docker 应用）

代理机

安装 Nginx + 证书工具

在代理机：

apt update
apt install -y nginx certbot python3-certbot-nginx
rm -f /etc/nginx/sites-enabled/default

代理机：先上“HTTP配置”（让 certbot 能签证书）

创建配置：

vim /etc/nginx/sites-available/web

贴入（先用 80，后面 certbot 会自动改成 443）：

# 静态站
server {
    listen 80;
    server_name static.example.com;

    location / {
        proxy_set_header Host $host;
        proxy_set_header X-Forwarded-For $remote_addr;
        proxy_set_header X-Forwarded-Proto $scheme;
        proxy_http_version 1.1;
        proxy_set_header Connection "";

        proxy_pass http://ORIGIN_IP:80;
    }
}

# docker 应用
server {
    listen 80;
    server_name app.example.com;

    location / {
        proxy_set_header Host $host;
        proxy_set_header X-Forwarded-For $remote_addr;
        proxy_set_header X-Forwarded-Proto $scheme;
        proxy_http_version 1.1;
        proxy_set_header Connection "";

        proxy_read_timeout 300s;
        proxy_send_timeout 300s;

        proxy_pass http://ORIGIN_IP:3000;
    }
}

检查并重启：

nginx -t
systemctl restart nginx

申请 HTTPS 证书并自动改 Nginx 到 443

在代理机执行（两个域名一起签）：

certbot --nginx -d static.example.com -d app.example.com

它会：

  • 自动加上 listen 443 ssl;
  • 自动把 80 做跳转到 https（你按提示选 Redirect）

DNS

把两个域名都解析到代理机 IP：

  • static.example.com -> PROXY_IP
  • app.example.com -> PROXY_IP

验证

在代理机跑：

curl -I https://static.example.com
curl -I https://app.example.com

代理机上看实时日志（排障很快）：

tail -f /var/log/nginx/access.log /var/log/nginx/error.log

1c1g 代理机

  • 只做转发 + TLS 终止，别上复杂 WAF/大日志落盘/压缩缓存
  • 并发多、SSE/长连接多时，代理机先升级到 2c2g 会明显稳

注意

为什么代理服务器要放在源服务器附近？

因为你的链路是：用户 → 代理 → 源站。代理和源站之间每次请求都要走一次。

  • 如果代理离源站远：每个请求都会多一次跨洲/跨网的 RTT，P95/P99 会炸，流式（SSE/WS）更明显。
  • 如果代理和源站同区域/同机房：代理→源站 RTT 可以做到毫秒级，代理只是“转发开销”，整体更稳定、可预测。

你可以把它理解成：

代理离用户近只能“看起来近”，但源站在美国就必须再跨一次；
代理离源站近则能把“中转的额外成本”压到最小。

遇到的问题

  • https://www.neoai.plus → 301 重定向死循环
  • https://neoai.plus → 504 Gateway Timeout

301 死循环（应用层）

根因：

  • 你同时在 代理机 和 源站 做了 http → https 跳转
  • 代理告诉源站：X-Forwarded-Proto: https
  • 源站看到“是 https”，又 return 301 https://...

于是形成闭环：

https://www → 代理 → 源站80 → 301到 https://www → 代理 → …

解决原则（非常重要）：

HTTPS 只在一层做。

既然是代理终止 HTTPS，源站就不再管 HTTPS。

504 超时（网络/防火墙层）

关键现象：

  • 源站本机 curl 127.0.0.1:3000 → 200 OK
  • iptables INPUT 已放行代理 IP → 3000
  • 但代理访问 23.133.52.53:3000 → 超时
  • tcpdump 显示：SYN 到了，但没有 SYN-ACK

真正的根因：

iptables -P FORWARD DROP
DOCKER-USER 链是空的

Docker 的流量 不是走 INPUT，而是走 FORWARD。

你放行的是 INPUT：

-A INPUT -s 代理IP --dport 3000 ACCEPT

但 Docker 返回包走的是：

FORWARD / DOCKER-USER

结果是：

  • 请求能进来
  • 回包被防火墙丢掉
  • 客户端看到的就是 504 / timeout

这是 Docker + iptables 的经典大坑。

最终修复

补齐：

iptables -I DOCKER-USER -m conntrack --ctstate ESTABLISHED,RELATED -j ACCEPT
iptables -I DOCKER-USER -s 代理IP -p tcp --dport 3000 -j ACCEPT
iptables -I DOCKER-USER -j RETURN
iptables -P FORWARD ACCEPT

于是：

  • SYN 能进
  • SYN-ACK 能回
  • 连接建立
  • 504 消失
