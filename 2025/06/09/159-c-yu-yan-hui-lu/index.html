// snprintf(char *str, size_t size, const char *format, ...);
    // 安全的格式化输出函数，它会将格式化的结果写入 str，最多写入 size-1 个字符，并自动添加 '\0'。
    // %d: 整数, %02X: 两位大写十六进制数 (不足两位前面补0), %s: 字符串。

TYPE_HEX=0x%02X：固定文本，%02X 会被第三个变量 type_code 的值替换。

  • X：表示以大写字母输出十六进制数。
  • 02：表示最少输出两位，如果不足两位，前面用 0 填充。例如，如果 type_code 是10（十六进制是A），会输出 0A；如果是255（十六进制是FF），会输出 FF。

strlen

用C函数strlen()获取字符串的长度。

C语言用\0标记字符串的结束

strlen()得出的字符串中的字符数包括空格和标点符号,sizeof给出的更大,由于将字符串末尾不可见的空字符也计算在内,所以一般使用sizeof都需要-1

strlen(data_line) 获取 data_line 字符串的长度 (不包括末尾的空字符 '\0')

atoi

atoi(const char *str); 将数字字符串转换为整数 (int)。

缺点是会将浮点数转变为整数，840.5->840

atoi (ASCII to Integer) 函数在 C 和 C++ 中用于将字符串转换为整数。它的转换规则如下：

  1. 跳过前导空白字符：例如空格、制表符等。
  2. 处理可选的正负号：+ 或 -。
  3. 读取连续的数字字符：’0’ 到 ‘9’。
  4. 在第一个非数字字符处停止：一旦遇到不属于数字的字符，它就会停止解析，并返回到目前为止已经成功转换的整数值。

⚠️:atoi与atof在严肃的程序里是不被推荐的由于(缺乏错误处理和会出现未定义协议)

f 后缀的意义: 在 C/C++ 中，浮点数字面量默认是 double (双精度浮点数) 类型。

  • 1240.0 是一个 double 类型的值（通常占8个字节）。
  • 1240.0f 是一个 float 类型的值（通常占4个字节）。

malloc

// 在任务函数内部定义
uint8_t *data_tmp = (uint8_t *)malloc(UART_BUF_SIZE);
uint8_t *data_tmp: 定义一个指向 uint8_t（无符号8位整数，即字节）类型的指针 data_tmp。

malloc(UART_BUF_SIZE): 从堆内存中动态分配一块内存。UART_BUF_SIZE 是一个预定义的宏，指定了分配内存的大小（例如 1024 字节）。

(uint8_t ): 强制类型转换，因为 malloc 函数返回的是一个通用的 void 指针。

作用: 这个 data_tmp 缓冲区用来临时存放从 UART 驱动的内部环形缓冲区（Ring Buffer）中读取出来的原始字节数据。我们一次性从驱动读取一批数据到这里，然后再逐字节地进行分析和处理，这样比一个字节一个字节地调用 uart_read_bytes 效率更高。

volatile

作用：这是一个标准的 C 语言关键字，但在中断和多任务编程中至关重要。它告诉编译器：“这个变量随时可能在你看不到的地方被改变（比如在中断里），所以请不要对它进行优化。每次使用它时，都必须老老实实地从内存中重新读取，每次修改它时，都必须立即写回内存。”

指针类型转换和结构体成员访问

const wifi_promiscuous_pkt_t *pkt = (wifi_promiscuous_pkt_t *)buf;
const uint8_t *payload = pkt->payload; // payload是数据包的实际内容
const int len = pkt->rx_ctrl.sig_len;  // len是数据包的长度
  • const wifi_promiscuous_pkt_t *pkt = (wifi_promiscuous_pkt_t *)buf;
    • buf: 通常，buf 是一个 void * 类型的指针，它指向从网卡混杂模式（promiscuous mode）捕获到的原始数据包的起始地址。混杂模式允许网卡接收所有流经它的数据包，而不仅仅是发给它自己的数据包。
    • (wifi_promiscuous_pkt_t *)buf: 这是一个强制类型转换 (type casting)。它告诉编译器，将 buf 这个 void * 指针所指向的内存区域，按照 wifi_promiscuous_pkt_t 结构体的定义来解释。
    • const wifi_promiscuous_pkt_t *pkt: 定义了一个名为 pkt 的常量指针，它指向一个 wifi_promiscuous_pkt_t 类型的结构体。const 关键字表示通过 pkt 指针不能修改它所指向的数据。
    • 作用： 这行代码的目的是将捕获到的原始数据 buf 解释为一个预定义的 wifi_promiscuous_pkt_t 结构体。这个结构体通常定义了 Wi-Fi 捕获数据包的通用格式，包括一些控制信息和实际的载荷数据。
  • const uint8_t *payload = pkt->payload;
    • pkt->payload: 这里使用了箭头操作符 ->。当 pkt 是一个指向结构体的指针时，pkt->payload 用于访问 pkt 所指向的 wifi_promiscuous_pkt_t 结构体内部的 payload 成员。
    • payload 成员通常是一个 uint8_t * 类型的指针，它指向数据包的实际内容，也就是 MAC 帧的头部和载荷部分。
    • 作用： 这一步从解析后的数据包结构体中提取出我们最关心的部分——数据包的实际有效载荷，并将其存储在一个 const uint8_t * 类型的指针 payload 中，方便后续处理。
  • **const int len = pkt->rx_ctrl.sig_len; **
    • pkt->rx_ctrl: 同样使用箭头操作符访问 pkt 指向的结构体中的 rx_ctrl 成员。rx_ctrl 通常是一个嵌套的结构体，包含接收到的数据包的各种控制信息，比如信号强度、数据包长度等。
    • rx_ctrl.sig_len: 访问 rx_ctrl 结构体中的 sig_len 成员。sig_len 通常代表了实际接收到的数据包的有效长度（信号长度）。
    • 作用： 获取数据包的长度，这在后续遍历 payload 时非常重要，可以避免越界访问。

分文件管理

分文件管理时，

在.c文件中定义,.h文件中extern声明,那么包含这个.h文件的其他.c文件就都可以使用声明过的函数或者变量了。

不可以在头文件中定义,因为包含头文件的源文件都会定义,会导致重定义的报错。

特殊情况（补充说明，但对你目前解决问题没有直接影响）：

  1. static 关键字：
    • 对于变量，static 放在全局作用域时，会使其具有内部链接属性。这意味着该变量只在定义它的那个 .c 文件内部可见，其他 .c 文件即使有同名的 static 变量，它们也是独立的互不影响的副本。
    • 这就是 ESP-IDF 推荐日志 TAG 的做法：每个 .c 文件都有自己的 static const char *TAG = "MODULE_NAME";，它们是独立的。
    • 对于函数，static 也会使其具有内部链接属性，函数只在当前 .c 文件内可见。
  2. inline 关键字：
    • 对于函数，inline 是一种提示编译器进行内联优化的关键字。为了使 inline 函数能在多个 .c 文件中被定义而不会出现重复定义错误，C++ 标准（以及 C99/C11 标准）规定 inline 函数允许多重定义，但所有定义必须完全相同。链接器会选择其中一个定义。这在头文件中定义小函数时常用。

union联合体

union 通用事件结构体的核心功能，就是用一种标准化的“容器”，将系统中各种不同类型、不同来源的事件/消息进行统一封装。

// 定义“标签”的种类
typedef enum {
    EVENT_FROM_NETWORK,  // 标签：网络来的
    EVENT_FROM_BUTTON,   // 标签：按键来的
    EVENT_FROM_SENSOR,   // 标签：传感器来的
} event_type_t;
// 一个包含所有可能数据类型的 "魔术内胆"
typedef union {
    network_command_t net_cmd;   // 网络指令（可能是一个结构体）
    button_press_t    btn_press; // 按键动作（可能是一个枚举）
    float             sensor_val;  // 传感器读数（可能是一个浮点数）
} event_payload_t;


// 最终的 "快递盒" 结构体
typedef struct {
    event_type_t      type;    // 外面的“标签”
    event_payload_t   payload; // 里面的“魔术内胆”
} universal_event_t;

基础算法

位运算

1.高位0x03，低位 0x36，合并十六位后转换为十进制

uint8_t concHigh = 0x03;
uint8_t concLow = 0x36;

// 组合成16位整数
uint16_t rawValue = (0x03 << 8) | 0x36;
                  = (3 << 8) | 54
                  = 768 | 54
                  = 822

0x36=3*16+6=54

位移运算 (3 << 8)

<< 是左移运算符，意思是把二进制往左移动 8 位。

3 的二进制是：

0000 0011

左移 8 位后：

0000 0011 0000 0000

这相当于：

3 × 2⁸ = 3 × 256 = 768

所以 (3 << 8) 的结果是 768 (十进制)，也就是 0x300 (十六进制)。

二进制转十进制：
从右数起，第 n 位如果是 1，就加上 2ⁿ。

所以：

0000 0011 0011 0110
= 2⁹ + 2⁸ + 2⁵ + 2⁴ + 2² + 2¹
= 822

为什么 (0x03 << 8) 等价于 (3 << 8)

因为两者的数值完全一样。
0x03 表示的数值就是十进制的 3。

左移 8 位只看它的二进制位，
而二进制相同 → 移位效果当然一样：

0000 0011   ← 0x03 或 3
左移8位 → 0000 0011 0000 0000

这结果在十进制里是：

3 × 2⁸ = 768

也可以写成十六进制：

0x03 << 8 = 0x300
