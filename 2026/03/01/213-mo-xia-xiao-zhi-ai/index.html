摸下小智AI
2026-04-20 12:02:24

背景

与clawbot一样，到处都是，所以摸一下

原理

  1. ESP32将麦克风数据通过i2s接收并进行回声处理后，使用opus编码将二进文件（俗称音频流）实时上传到服务器，并将LLM结果返回后通过opus解码后I2S播放，如果要调用外设，比如开关灯，需要通过mcp
  2. MCP，给模型用的接口，ESP32与服务器建立websocket连接口，ESP32会发个hello，确认连接完成，然后模型可以发送tools/list，来获取当前支持的MCP，MCP格式有工具名，描述，参数等，可以用来实现开关灯

📌claude对我理解的补充说明

关于第1点的细节补充：

  • “二进文件”更准确说是二进制数据包（不是文件，是实时流）
  • 完整流程应该是：I2S接收 → AudioProcessor(回声处理) → Opus编码 → WebSocket上传
  • 下行是：WebSocket接收 → Opus解码 → I2S播放
  • I2S是数字音频接口，连接ESP32和音频芯片（如ES8311）

关于第2点的细节补充：

  • MCP完整流程：hello → initialize → tools/list → tools/call
  • 注意：不是”模型发送tools/list”，而是服务器代表模型发送
  • MCP基于JSON-RPC 2.0协议，每个消息都有jsonrpc、method、params、id字段

我吸收的

  1. 音频流is what

    opus编码后的二进制文件

  2. 为什么需要AEC回声消除

    为了消除麦克风采集到的扬声器数据,消除回音

  3. 队列的作用

    用于不同任务之间的缓冲

📌 claude对我理解的扩展（保留你的原文，下面是补充）

关于”音频流”：

  • ✅ 你说的对，就是opus编码后的二进制数据
  • 补充：更准确说是数据包流，每个包60ms（960个采样点@16kHz），不是一个完整文件

关于”AEC回声消除”：

  • ✅ 核心理解正确
  • 补充：AEC通过对比扬声器参考信号和麦克风输入，计算出回声部分并减去

关于”队列”：

  • ✅ 完全正确
  • 补充：项目中有4个队列，audio_encode_queue_连接AudioProcessor→OpusEncoder

三任务架构

AudioInputTask (优先级8)  - 从麦克风读取数据，优先级最高防止丢帧
OpusCodecTask  (优先级5)  - 编解码处理
AudioOutputTask(优先级4)  - 播放音频，优先级最低

关键数据流

四个队列：

  • audio_encode_queue_：等待编码的PCM数据
  • audio_send_queue_：编码后等待发送的Opus数据
  • audio_decode_queue_：接收到等待解码的Opus数据
  • audio_playback_queue_：解码后等待播放的PCM数据

采样率转换

为什么需要重采样：

  • 音频芯片可能是48kHz
  • AudioProcessor和Opus都需要16kHz
  • input_resampler_负责转换：48kHz → 16kHz

计算公式：

采样点数 = 采样率 × 时间
16kHz × 0.06s = 960个采样点（Opus一帧）
48kHz × 0.01s = 480个采样点（AudioInputTask每次读取）

实际应用场景

用户说：”帮我开灯”

1. 麦克风采集 → I2S → AudioProcessor → Opus编码 → WebSocket上传
2. 服务器ASR识别："帮我开灯"
3. LLM理解意图，查找MCP工具列表，找到gpio.write
4. 服务器发送：tools/call {name: "gpio.write", arguments: {pin: 5, value: 1}}
5. ESP32执行GPIO控制，点亮LED
6. 服务器TTS生成："好的，已为您开灯"
7. 音频流下发 → Opus解码 → I2S播放

易错点总结

  1. ❌ 音频流是MP3 → ✅ 是Opus，延迟更低
  2. ❌ MCP方法是tools/execute → ✅ 是tools/call
  3. ❌ Opus配置参数包括码率 → ✅ 是采样率、声道数、帧时长

调试技巧

查看音频数据流：

// 在AudioInputTask添加日志
ESP_LOGI(TAG, "Queue depth: %d", uxQueueMessagesWaiting(audio_encode_queue_));

检查MCP工具调用：

// 在mcp_server.cc的ExecuteTool添加日志
ESP_LOGI(TAG, "Calling tool: %s with args: %s", tool_name, args_json);

监控任务状态：

# 串口监视器查看关键日志
idf.py monitor | grep -E "AudioService|McpServ
