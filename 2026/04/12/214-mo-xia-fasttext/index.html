摸下fasttext
2026-04-20 12:02:24 #fasttext #NLP

起因

QQ群里每天几十条用户提问，”codex怎么配置”、”403报错怎么办”、”有什么套餐”……重复问题占了大半。人工回复效率低，正则匹配维护成本高。想要一个自动给出对应回复的系统，当前已有的是模糊关键词匹配，缺点是命中率一般且容易误判，优点是速度快。

为什么不上 LLM？群消息要求快、准、确定性强——LLM 可能回复慢、不稳定，主要是我还没有掌握。FastText 推理 < 100ms，分类结果确定，适合这个场景。

FastText 是什么

文本分类模型，通过将语句进行分类，分词处理后，转化为词向量进行训练，完成后可以对新的语句进行判断分类。

核心流程：分词 → 词向量 → 平均池化 → 神经网络 → 分类概率。

实战

收集数据

足够多的数据，我使用的来自于qq群聊。

数据标注

使用python脚本将文本按照__label__类型 具体语句整理为数据集。

__label__提问 codex怎么配置
__label__闲聊 哈哈哈哈

一行一条，__label__类别 空格 文本内容。

模型训练

将整理好的数据集导入训练即可。

import fasttext

model = fasttext.train_supervised(
    input='train.txt',
    epoch=50,
    lr=0.5,
    wordNgrams=2,
    dim=100,
    loss='softmax'
)
model.save_model('chat_model.bin')

推理

model = fasttext.load_model('chat_model.bin')
result = model.predict(' '.join(jieba.cut('codex怎么配置')))
# → ('__label__提问', 0.78)

中文需要先 jieba 分词，英文不需要。

从关键词匹配到模型

一开始我用的是模糊关键词匹配——消息里出现”codex”就回复codex相关内容，出现”403”就回复403解决方案。简单直接，但问题是一时之间无法将所有排列组合的方式想清楚，用户问法千变万化，”cx怎么搞”、”报错403咋整”这些表达很难穷举。

所以想到用模型来补位：让 FastText 先判断”这条消息是不是在提问”，是提问再走关键词意图路由。模型只做一件事——过滤掉闲聊和确认消息，避免误触发回复。具体的回复匹配还是交给关键词，因为关键词规则透明可调，改个配置就行，不用重训模型。

消息 → [模型:是否提问] → [意图路由:关键词评分] → 回复

模型负责”是不是提问”这一个判断，关键词负责”是什么问题”。各司其职。

灵光一闪：词向量 vs Token

训练的时候突然想到——词向量把”配置”变成 [−0.3, 0.7, 0.1, ...]，LLM 的 token embedding 也是把文本变成数字。它们是不是同一回事？

查了一下，本质上确实是同一回事——都是把离散的文本映射到连续的数字空间，让语义相近的词在空间中距离更近。

区别在于：

词向量（FastText） Token（GPT）
粒度 完整词 子词（可能拆开）
维度 100维 12288维
性质 静态（训练后固定） 动态（随上下文变化）

词向量是 token embedding 的早期形态。技术演进：词向量(2013) → FastText(2016) → BERT(2018) → GPT(2020+)。

核心没变：把语言变成数字，让机器能算。
